# 05 - Decisions: החלטות ארכיטקטוניות (ADR)

## ADR-001: PWA במקום Electron

**תאריך**: 2026-04-12  
**סטטוס**: הוחלף חלקית על ידי ADR-010

**הקשר**: נדרשת אפליקציה פורטבילית שלא תיחסם ע"י Windows SmartScreen או אנטי-וירוס.

**החלטה**: PWA (Progressive Web App) במקום Electron.

**נימוק**:
- Electron מפיק קובץ EXE שנחסם ע"י SmartScreen ללא Code Signing Certificate (~$200-400/שנה)
- PWA מתנהגת כאפליקציה עצמאית (חלון נפרד, אייקון בשולחן עבודה)
- לא דורשת התקנה מסורתית
- עובדת אופליין דרך Service Worker
- נתונים נשמרים מקומית ב-IndexedDB

**חסרונות**: IndexedDB יכול להימחק בניקוי דפדפן. פתרון: עידוד ייצוא תקופתי + אזהרה.

**עדכון 2026-06-15**: היעד הסופי עבר לאפליקציית Desktop דרך Tauri. שכבת ה-PWA נשארת שימושית כאופן בנייה/אופליין של ה-frontend, אך אינה דרך ההפצה העיקרית.

## ADR-002: @hebcal/core כספריית לוח עברי

**תאריך**: 2026-04-12  
**סטטוס**: מאושר

**הקשר**: כל החישובים מבוססים על לוח עברי. צריך ספרייה אמינה.

**החלטה**: שימוש ב-`@hebcal/core`.

**נימוק**:
- הספרייה הכי בוגרת ומתוחזקת ל-JavaScript
- תומכת בכל סוגי השנים (מעוברת, חסרה, מלאה, כסדרה)
- המרה דו-כיוונית עברי↔לועזי
- תחזוקה פעילה, קהילה גדולה

**הערה**: נעטוף אותה בשכבת abstraction (`hebrewDate.ts`) כדי שהמנוע ההלכתי לא יהיה תלוי ישירות.

## ADR-003: ספירת ימים כוללנית (Inclusive)

**תאריך**: 2026-04-12  
**סטטוס**: מאושר

**הקשר**: סעיף כ"א בהלכות קובע ששני הימים (ראיה ראשונה + ראיה שנייה) נספרים.

**החלטה**: `hebrewDaysBetween(a, b) = b - a + 1`

**דוגמה**: א' ניסן → כ"ה ניסן = 25 ימים (לא 24).

**נימוק**: זו הדרישה ההלכתית המפורשת. טעות בנקודה זו תגרום לחישוב שגוי של כל ההפלגות.

## ADR-004: סטטוס "Dormant" לווסתות שנעקרו

**תאריך**: 2026-04-12  
**סטטוס**: מאושר

**הקשר**: סעיף צ"ה — ווסת קבוע שנעקר חייב להיזכר לעולם (עד שנקבע ווסת חדש).

**החלטה**: VesetRecord יכלול סטטוס `'dormant'` בנוסף ל-`'fixed'`, `'non-fixed'`, `'uprooted'`.

**נימוק**: ווסת dormant אינו פעיל (לא יוצר חששות) אבל נבדק בכל ראיה חדשה — אם הראיה מתאימה לו, הוא חוזר כקבוע. בלי סטטוס זה, נאבד מידע קריטי.

**חריגים**: לא חל על ווסת מעין פתוח (ספק ווסת), שבוע, דילוג, סירוג — סעיפים צז-צח.

## ADR-005: פסק הלכתי עיקרי בלבד, ללא הצגת חומרות

**תאריך**: 2026-04-12  
**סטטוס**: מאושר

**הקשר**: קובץ ההלכות מביא פסק עיקרי ולעיתים מוסיף "יש מחמירים" או "במקום הצורך תשאל רב".

**החלטה**: האפליקציה עוקבת אחרי הפסק העיקרי **בלבד** ולא מציגה את החומרה למשתמש. "שאל רב" מוצג רק כשאין פסק חד-משמעי.

**נימוק**: 
- המשתמש דורש ודאות, לא ספקות
- הצגת חומרות תבלבל משתמשים עם ידע בסיסי
- הפסק העיקרי בקובץ הוא מה שנפסק להלכה

**יישום**: טבלת מיפוי ספקות ב-`03-specification.md` מגדירה בדיוק מה נלקח בכל סעיף.

## ADR-006: ווסתות מורכבים → "שאל רב"

**תאריך**: 2026-04-12  
**סטטוס**: מאושר

**הקשר**: סעיפים פ"ז (ווסת גוף מורכב) וצ"ג (ווסת קפיצות מורכב) אומרים "אינם מצויים, יש פרטים רבים, יש לשאול רב".

**החלטה**: האפליקציה תזהה מצבים שעשויים להצביע על ווסת מורכב ותציג הודעת "שאל רב" — בלי לנסות לחשב.

**זיהוי**: אם נמצאת חפיפה בין ווסת גוף/קפיצות לבין ווסת חודש/הפלגה (שניהם מתאימים), מוצגת אזהרה.

## ADR-007: IndexedDB + ייצוא JSON

**תאריך**: 2026-04-12  
**סטטוס**: מאושר

**הקשר**: PWA לא יכולה לכתוב ישירות לקבצים. צריך אחסון מקומי מתמיד.

**החלטה**: IndexedDB כאחסון ראשי + ייצוא/ייבוא JSON ידני.

**נימוק**:
- IndexedDB: מתמיד, מהיר, תומך במבנים מורכבים, ללא הגבלת גודל מעשית
- JSON ייצוא: גיבוי פשוט, ניתן להעברה בין מכשירים, קריא
- `idb` library: wrapper פשוט שהופך IndexedDB ל-Promise-based

**סיכון**: ניקוי נתוני דפדפן מוחק IndexedDB.  
**מיטיגציה**: אזהרה בולטת + תזכורת חודשית לייצוא + אזהרה לפני מחיקת נתונים.

## ADR-008: React Context + useReducer (לא Redux)

**תאריך**: 2026-04-12  
**סטטוס**: מאושר

**הקשר**: צריך state management לנתוני הראיות, תוצאות החישוב, והגדרות.

**החלטה**: React Context + useReducer.

**נימוק**:
- אין צורך ב-Redux — אין middleware מורכב, אין async flows רבים
- הנתונים זורמים בכיוון אחד: IndexedDB → Context → Components
- פשטות > מורכבות בפרויקט עם UI לא מורכב במיוחד

## ADR-009: תאריך עברי משתנה בשקיעה — חישוב לפי מיקום

**תאריך**: 2026-04-18  
**סטטוס**: מאושר

**הקשר**: התאריך העברי (וחלוקת עונות יום/לילה) תלוי בחמה, לא בחצות האזרחי. ראיה בשעה 19:00 בירושלים בקיץ היא יום, אבל בחורף היא לילה — וגם התאריך העברי כבר התחלף.

**החלטה**:
1. נעשה שימוש ב-`@hebcal/core` Zmanim class לחישוב זמני שקיעה/הנץ מדויקים.
2. **חוק**: לפני השקיעה → עונת יום + תאריך עברי של היום האזרחי. מהשקיעה והלאה → עונת לילה + תאריך עברי הבא.
3. **אין אזור אפור** של בין השמשות בקוד — השקיעה היא חיתוך נקי לפי הפסק: "ראיה ביום תלוי בחמה, וכל זמן החמה עד השקיעה נחשב כיום. בין השמשות שכבר אין חמה נראית לעין נחשב כלילה".
4. רשימת ערים מובנית + אפשרות מיקום מותאם (lat/lng/timezone).
5. ברירת מחדל: ירושלים.
6. המשתמש יכול לעקוף את החישוב האוטומטי אם הוא יודע את העונה בוודאות.

**נימוק**:
- פסק הלכתי ברור מהמשתמש — אין צורך בטיפול בספק בין השמשות
- תאריך אזרחי במחשב לא מייצג תאריך עברי הלכתי נכון
- שעת ראיה מדויקת + מיקום + @hebcal → תוצאה הלכתית מדויקת

**חסרונות**: דורש מהמשתמש להזין מיקום ושעת ראיה. פתרון: auto-detection ברירת מחדל לירושלים + טופס פשוט.

## ADR-010: הפצה שולחנית באמצעות Tauri

**תאריך**: 2026-06-15  
**סטטוס**: מאושר

**הקשר**: היעד המעשי הוא מוצר שולחני שהמשתמש מפעיל כמו אפליקציית Windows רגילה, בלי לפתוח דפדפן ובלי לבקש מסוכן פיתוח להריץ שרת Vite.

**החלטה**: לשמור את React/Vite כשכבת frontend, אך לארוז אותה באמצעות Tauri 2 לאפליקציית Desktop.

**נימוק**:
- אין צורך לכתוב מחדש את מנוע החישוב או את ה-UI.
- Tauri מייצר אפליקציה קטנה בהרבה מ-Electron.
- האפליקציה רצה בחלון עצמאי ולא בכתובת `localhost`.
- נתוני המשתמש נשארים מקומיים.
- אפשר לבנות Windows installer ב-GitHub Actions גם אם במחשב המקומי חסרים Rust/C++ Build Tools.

**השלכות**:
- לבנייה מקומית נדרשים Rust/Cargo ו-Microsoft C++ Build Tools.
- ללא Code Signing Certificate ייתכן ש-Windows SmartScreen יציג אזהרה על קובץ ההתקנה.
- IndexedDB נשאר אחסון מקומי ראשי בשלב זה; מעבר ל-SQLite/קובץ מקומי יישקל רק אם נצטרך שליטה טובה יותר בגיבוי ובמיקום הנתונים.
